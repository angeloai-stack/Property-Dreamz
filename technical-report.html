01 — Descripción General

Descripción General del Proyecto

Property Dreamz es una plataforma de bienes raíces diseñada para conectar compradores americanos con desarrollos inmobiliarios mexicanos verificados. Se posiciona como el único portal donde cada propiedad está certificada antes de ser publicada, bajo el sello CM/RE – Certified Mexico Real Estate.

14Next.js
35+Componentes
8Páginas App
4API Routes
Estado actual: La UI está completamente construida como MVP funcional. Se cuenta con 5 formularios interactivos con captura de leads y 4 API routes. Los datos de propiedades son estáticos (mock). Las integraciones con Sanity CMS, CRM y Mapbox están declaradas pero aún no conectadas.

Patrón de máquina de estados

Todos los formularios implementan el mismo patrón con useState:

type Status = "idle" | "submitting" | "success" | "error"

const [status, setStatus] = useState<Status>("idle")
const [fields, setFields] = useState<Fields>(INIT)

# set() — helper genérico para actualizar cualquier campo
const set = (k: keyof Fields) => (e: ChangeEvent<...>) =>
  setFields(p => ({ ...p, [k]: e.target.value }))

# Submit — async fetch con transición de estado
async function handleSubmit(e: FormEvent) {
  e.preventDefault()
  setStatus("submitting")
  try {
    const res = await fetch("/api/...", { method: "POST", ... })
    setStatus(res.ok ? "success" : "error")
  } catch {
    setStatus("error")
  }
}

09 — Explore Map

Módulo Explore Map

Es el módulo más complejo del proyecto. Combina tres componentes coordinados por estado local en el page component.

Arquitectura del módulo

explore-map/page.tsx (Client Component) orquesta el estado. Los subcomponentes reciben props y callbacks — no tienen estado propio.

Estado gestionado en page.tsx

EstadoTipoPropósito
activePinnumber | nullSincroniza mapa y lista (card activa = pin resaltado)
typeFilter"All" | "House" | "Condo" | "Land"Filtro de tipo de propiedad
currency"USD" | "MXN"Moneda de visualización de precios
sortBy"relevance" | "price_asc" | "price_desc"Ordenamiento de resultados
verifiedOnlybooleanFiltrar solo propiedades verificadas
searchValstringBúsqueda de texto libre
mobileView"map" | "list"Toggle de vista en mobile

Filtrado y ordenamiento

Procesado con useMemo para evitar recálculos innecesarios:

listings
  → filter(searchVal)      # match en título, zona
  → filter(typeFilter)     # tipo de propiedad
  → filter(verifiedOnly)   # solo verificadas
  → sort(sortBy)           # precio asc/desc o relevancia
  → filtered[]             # resultado para ListingCards + conteo

Formateadores de precio (utils.ts)

  • formatMXN(value) → $12M MXN / $850K MXN
  • formatUSD(value) → USD $1.2M / USD $450K
  • formatPrice(mxn, usd, currency) → Dual-currency formatter
10 — Patrones Técnicos

Patrones Técnicos Clave

Server vs Client Components

El proyecto usa correctamente el App Router de Next.js 14: los componentes son Server Components por defecto. Se marca "use client" solo donde hay interactividad real (Navbar, explore-map/page.tsx, todos los formularios de src/components/forms/).

Gestión de Estado

  • Estado local con useState — apropiado para el scope actual
  • useMemo para el filtrado/ordenamiento de listings en explore-map
  • Máquina de estados en formularios: idle → submitting → success | error
  • Sin Zustand, Redux ni Context API — la aplicación no lo requiere aún

11 — Integraciones

Integraciones Declaradas (Pendientes)

Las API routes están creadas y funcionales. El siguiente paso es conectarlas a servicios externos.

HubSpot CRM
Las 4 API routes reciben los datos de los formularios. Falta instanciar @hubspot/api-client con HUBSPOT_API_KEY para crear contacts/deals automáticamente.
Resend / Email
Cada form submission debe disparar: (1) notificación al equipo de ventas, (2) email de confirmación al usuario. Requiere RESEND_API_KEY.
Sanity CMS
Para gestión dinámica de propiedades. Sustituirá el array data.ts. Depende de: SANITY_PROJECT_ID, SANITY_DATASET.
Mapbox GL
El mapa SVG actual es decorativo. Mapbox reemplazará MapPanel.tsx con mapa real. Depende de: MAPBOX_TOKEN.
Progreso de formularios: Los 5 formularios tienen UI funcional, validación HTML5, estados de submit/success/error, y API routes correspondientes. Solo falta conectar las routes con HubSpot y Resend.
12 — Métricas de Calidad

Métricas de Código y Calidad

Lo que está bien implementado

  • TypeScript con strict mode — sin tipos any en la capa UI
  • Separación clara Server / Client components
  • Sistema de diseño coherente con tokens reutilizables
  • Tailwind v4 — sintaxis canónica aplicada en todos los archivos
  • 5 formularios con máquina de estados completa (idle/submitting/success/error)
  • 4 API routes creadas y listas para conectar a CRM
  • Privacy consent obligatorio en todos los formularios (requerimiento legal México)
  • JSON-LD estructurado: Organization + WebSite + FAQPage
  • robots.ts y sitemap.ts generados automáticamente
  • Componentes accesibles con ARIA labels y navegación por teclado
  • Mobile-first con breakpoints consistentes
  • Barrel exports organizados (ui/, layout/, forms/)
  • Co-location de datos por feature

Áreas de mejora identificadas

  • API routes sin integración real (CRM + email pendientes)
  • Datos de propiedades completamente estáticos (sin Sanity ni API)
  • Mapa SVG decorativo sin funcionalidad real (sin Mapbox)
  • Página /saved es stub vacío (requiere auth)
  • Página /properties sin grid de listings
  • WhatsApp link con número placeholder (5210000000000)
  • Imágenes de Unsplash como placeholders
  • Sin tests unitarios ni E2E
  • Sin manejo de errores (Error Boundary, error.tsx)
  • Sin loading states (loading.tsx por ruta)
  • Formularios usan useState nativo — se puede migrar a react-hook-form (ya instalado)
